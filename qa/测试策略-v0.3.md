## 五、需求与接口契约对齐结果

> v0.1 中提出的 Q1-Q6 已由 DR-003~DR-006 大部分拍板解决。v0.3 新增 O-8~O-11 四个文档冲突。

### 已确认决策汇总

| 决策项 | 结论 | 来源 |
|---|---|---|
| 初始体重基准 | 入组时一次性录入，全程不可修改 | DR-001-5 |
| 减重百分比公式 | `(初始 - 当日最新) / 初始 × 100%`，保留两位小数 | DR-001-5 |
| 负值处理 | 允许负值（增重），如实计入，不截断 | DR-001-5 |
| 计算时机 | 每日打卡后实时更新，取当日最后一次有效打卡体重 | DR-001-5 |
| **K 值** | **K=1（线性映射），减重百分比直接映射分数** | **DR-005 ✅ 已锁定** |
| 积分精度 | 存储 4 位浮点，展示 2 位四舍五入 | DR-003 Q1 |
| 并列排名 | 不跳号（1,1,2） | DR-003 Q2 |
| 结束日打卡 | 结束日当天 23:59 前有效 | DR-003 Q3 |
| 积分更新频率 | T+1，次日 00:05 | DR-003 Q4 |
| 初始体重修改 | 不允许自助修改，需联系产品组长人工处理 | DR-003 Q5 |
| 中途退出 | 标记"已退出"，不参与排名和结算；历史数据保留至 PK 结束 | DR-003 Q6 |
| 房主退出 | 不可退出，需解散 PK 房间 | DR-003 Q6 |
| 积分分母 | 三个维度统一用"用户加入后实际经过天数" | DR-006 Q8 |
| 房间解散 | 状态变为 DISSOLVED，只读 30 天后清理，不触发结算 | DR-006 Q13 |
| 结算触发 | PK 到期日 00:00 自动触发结算定时任务 | DR-002 |
| 结束体重取值 | 取最后一次有效打卡的体重 | DR-002 / DR-006 |
| 拍照防作弊 | 方案 A：前端 Canvas 水印 + 禁相册（L1），L2 迭代服务端校验 | 菲菲拍板 |
| 水印内容 | 时间（精确到分钟）+ 用户昵称（脱敏），不含地理位置 | 菲菲拍板 |
| 结算快照独立性 | 结算任务从原始数据全量重算，与 T+1 日常更新独立 | Q9 菲菲确认 |

### 6.5 积分计算（M5 核心 — P0 必须双人复核）

> ✅ **K=1 已锁定（DR-005）**。以下预期值按 K=1 线性映射计算。
>
> **积分公式（当前已确认部分）：**
> ```
> 综合积分（满分 100）= 
>   减重得分 + 打卡得分 + 饮食得分 + 运动得分
>
> 减重得分 = min(减重百分比, 100) × 0.70
>   减重百分比 = (初始体重 - 最新体重) / 初始体重 × 100
>   ⚠️ "最新体重"定义有冲突：PRD=单次最新，DR-004=近3天均值（O-8）
>
> 打卡得分 = (已打卡天数 / 加入后实际经过天数) × 100 × 0.15
> 饮食得分 = (饮食打卡天数 / 加入后实际经过天数) × 100 × 0.08
> 运动得分 = (运动打卡天数 / 加入后实际经过天数) × 100 × 0.07
>
> 存储 4 位浮点，展示 2 位四舍五入
> ```

> ⚠️ **以下用例按 PRD 公式（单次体重 + K=1）计算。若菲菲确认用 DR-004 的 3 天均值，需重新计算。**

| 用例ID | 场景 | 预期（K=1 手算） | 优先级 |
|---|---|---|---|
| TC-SCORE-001 | 减重 10%（75→67.5kg），21天全勤，饮食运动全✅ | 减重分=min(10,100)×0.7=**7.00**；打卡=100×0.15=**15.00**；饮食=100×0.08=**8.00**；运动=100×0.07=**7.00**；**总分=37.00** | P0 |
| TC-SCORE-002 | 减重 5%（80→76kg），21天全勤，饮食运动全✅ | 减重分=min(5,100)×0.7=**3.50**；打卡=**15.00**；饮食=**8.00**；运动=**7.00**；**总分=33.50** | P0 |
| TC-SCORE-003 | 增重（75→77kg），全勤打卡 | 减重%=(75-77)/75×100=-2.667；减重分=min(-2.667,100)×0.7=**-1.87**（若允许负分）；⚠️ O-2：若下限0则减重分=0；**总分=28.13（负分）或 30.00（下限0）** | P0 |

## 九、⚠️ K=1 权重失衡风险分析

> DR-005 已锁定 K=1（线性映射），但测试侧发现这可能导致**产品设计的 70% 减重权重名存实亡**。

### 问题说明

| 维度 | 权重 | 满分条件 | 现实场景得分 |
|---|---|---|---|
| 减重 | 70% | 减重 100%（不可能） | 减重 5% → 3.5 分；减重 10% → 7 分 |
| 打卡 | 15% | 全勤 | 全勤 → 15 分 |
| 饮食 | 8% | 全勤 | 全勤 → 8 分 |
| 运动 | 7% | 全勤 | 全勤 → 7 分 |

### 具体案例

**用户 A**：减重 5%（80→76kg），全勤打卡、饮食运动全✅
- 减重分 = 5 × 0.7 = **3.50**
- 打卡分 = 100 × 0.15 = **15.00**
- 饮食分 = 100 × 0.08 = **8.00**
- 运动分 = 100 × 0.07 = **7.00**
- **总分 = 33.50**

**用户 B**：体重持平，全勤打卡、饮食运动全✅
- 减重分 = 0
- 打卡分 = 15.00
- 饮食分 = 8.00
- 运动分 = 7.00
- **总分 = 30.00**

**结论**：用户 A 减重 5% 只比用户 B（零减重）多 3.5 分。打卡维度（30 分）在实际得分中远超减重维度。**70% 的减重权重在 K=1 下几乎没有区分度。**

### 测试侧建议

1. 此问题属于**产品规则设计**范畴，测试侧不代 PM 做决策
2. 但会在测试报告中标注：**"K=1 下减重维度区分度极低，建议产品评估是否调整 K 值"**
3. 如果菲菲/v_juvnliu 认为可接受（"先跑起来，后续调"），测试侧按 K=1 执行即可
4. 如果决定调整 K 值，TC-SCORE-001~014 需全部重算
