## 一、测试目标

**一句话**：确保 v_juvnliu 和朋友能在 08-01 前完整跑通一场 PK 闭环（发起→打卡→排行榜→结算→海报），核心链路无 P0/P1 阻塞，积分计算绝对准确。

| 目标层次 | 具体标准 |
|---|---|
| L0 冒烟 | 登录→发起PK→加入PK→每日打卡→排行榜→结算→海报 7 步主链路可走通 |
| L1 功能 | M1～M7 全部 Must 功能通过，无 P0/P1 遗留 |
| L2 边界 | 积分计算精度、PK 周期临界点、体重输入边界值、并发打卡等场景覆盖 |
| L3 合规 | 单独同意弹框、免责声明、数据删除入口、隐私字段不泄露 |
| L4 体验 | 2-5 人真实场景模拟跑一场完整 PK，无明显卡顿/闪退/数据异常 |

## 三、风险清单

### P0 — 计算 / 数据安全

| # | 风险 | 影响 | 应对 | 状态 |
|---|---|---|---|---|
| P0-1 | **积分计算精度错误**（浮点精度、百分比截断/四舍五入） | 排行榜排名错误，动摇 PK 公信力 | 用例覆盖边界值+并列排名; Excel 手算独立复核; 结算结果双人验证 | ✅ K=1 已锁定 |
| P0-2 | **结算逻辑错误**（最终体重取值、天数 off-by-one） | 结算排名和海报数据错误 | DR-006：取最后一次有效打卡体重; 验证幂等性 | ✅ 规则已定 |
| P0-3 | **隐私泄露**（排行榜/接口返回绝对体重、照片可被他人访问） | 违反 PIPL，用户信任崩塌 | 接口层验证 response 不含绝对体重; 照片 fileID 访问权限校验 | 🔴 高优先级 |
| P0-4 | **打卡幂等性失效**（同日多次提交积分重复累加） | 积分虚高，排行榜失真 | 并发打卡测试; DB 唯一索引验证 | ⚠️ O-3/O-10 规则冲突 |
| P0-5 | **🔴 减重公式存在两个版本**（PRD 单次体重+封顶 vs DR-004 三天均值+无封顶） | 积分结果完全不同 | 已同步菲菲，要求统一口径（O-8） | 🔴 阻塞用例 |
| P0-6 | **🟡 K=1 导致权重失衡**（见第九节） | 减重 70% 权重名存实亡 | 验证实际排名是否符合产品预期 | 🟡 需产品确认 |

## 五、需求与接口契约对齐

### 5.1 已确认决策汇总

| 决策项 | 结论 | 来源 |
|---|---|---|
| 初始体重基准 | 入组时一次性录入，全程不可修改 | DR-001-5 |
| 减重百分比公式 | `(初始 - 当日最新) / 初始 × 100%`，保留两位小数 | DR-001-5 |
| 负值处理 | 允许负值（增重），如实计入，不截断 | DR-001-5 |
| 计算时机 | 每日打卡后实时更新，取当日最后一次有效打卡体重 | DR-001-5 |
| **K 值** | **K=1（线性映射），减重百分比直接映射分数** | **DR-005 ✅ 已锁定** |
| 积分精度 | 存储 4 位浮点，展示 2 位四舍五入 | DR-003 Q1 |
| 并列排名 | 不跳号（1,1,2） | DR-003 Q2 |
| 结束日打卡 | 结束日当天 23:59 前有效 | DR-003 Q3 |
| 积分更新频率 | T+1，次日 00:05 | DR-003 Q4 |
| 初始体重修改 | 不允许自助修改，需联系产品组人工处理 | DR-003 Q5 |
| 中途退出 | 标记"已退出"，不参与排名和结算；历史数据保留至 PK 结束 | DR-003 Q6 |
| 房主退出 | 不可退出，需解散 PK 房间 | DR-003 Q6 |
| 积分分母 | 三个维度统一用"用户加入后实际经过天数" | DR-006 Q8 |
| 房间解散 | 状态变为 DISSOLVED，只读 30 天后清理，不触发结算 | DR-006 Q13 |
| 结算触发 | PK 到期日 00:00 自动触发结算定时任务 | DR-002 |
| 结束体重取值 | 取最后一次有效打卡的体重 | DR-002 / DR-006 |
| 拍照防作弊 | 方案 A：前端 Canvas 水印 + 禁相册（L1），L2 迭代服务端校验 | 菲菲拍板 |
| 水印内容 | 时间（精确到分钟）+ 用户昵称（脱敏），不含地理位置 | 菲菲拍板 |
| 结算快照独立性 | 结算任务从原始数据全量重算，与 T+1 日常更新独立 | Q9 菲菲确认 |

### 7.5 积分计算（M5 核心 — P0 必须双人复核）

> **积分公式（K=1 已锁定，DR-005）：**
>
> ```
> 综合积分（满分 100）= 减重得分 + 打卡得分 + 饮食得分 + 运动得分
>
> 减重得分 = min(减重百分比, 100) × 0.70
>   减重百分比 = (初始体重 - 最新体重) / 初始体重 × 100
>   ⚠️ "最新体重"定义有冲突：PRD=单次最新，DR-004=近3天均值（O-8，待菲菲确认）
>
> 打卡得分 = (已打卡天数 / 加入后实际经过天数) × 100 × 0.15
> 饮食得分 = (饮食打卡天数 / 加入后实际经过天数) × 100 × 0.08
> 运动得分 = (运动打卡天数 / 加入后实际经过天数) × 100 × 0.07
>
> 存储 4 位浮点，展示 2 位四舍五入
> ```

> ⚠️ **以下用例按 PRD 公式（单次体重 + K=1）手算。若菲菲确认用 DR-004 的 3 天均值，需重新计算。**

## 九、K=1 权重失衡风险分析

> DR-005 已锁定 K=1（线性映射），但测试侧发现这可能导致**产品设计的 70% 减重权重名存实亡**。

### 问题说明

| 维度 | 权重 | 满分条件 | 现实场景得分 |
|---|---|---|---|
| 减重 | 70% | 减重 100%（不可能） | 减重 5% → 3.5 分；减重 10% → 7 分 |
| 打卡 | 15% | 全勤 | 全勤 → 15 分 |
| 饮食 | 8% | 全勤 | 全勤 → 8 分 |
| 运动 | 7% | 全勤 | 全勤 → 7 分 |

### 具体案例对比

**用户 A**：减重 5%（80→76kg），全勤打卡+饮食+运动
- 减重分 = 5 × 0.7 = **3.50**；打卡分 = **15.00**；饮食分 = **8.00**；运动分 = **7.00**；**总分 = 33.50**

**用户 B**：体重持平，全勤打卡+饮食+运动
- 减重分 = 0；打卡分 = **15.00**；饮食分 = **8.00**；运动分 = **7.00**；**总分 = 30.00**

**结论**：用户 A 减重 5% 只比用户 B（零减重）多 3.5 分。打卡+饮食+运动（30 分）在实际得分中远超减重维度，70% 的减重权重在 K=1 下几乎没有区分度。

### 测试侧立场

1. 属于**产品规则设计**范畴，测试侧不代 PM 做决策
2. 测试报告中会标注：**"K=1 下减重维度区分度极低，建议产品评估是否调整 K 值"**
3. 如果菲菲/v_juvnliu 认为可接受（"先跑起来，后续调"），测试侧按 K=1 执行
4. 如果决定调整 K 值，TC-SCORE-001~015 需全部重算
