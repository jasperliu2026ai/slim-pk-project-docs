## 一、鉴权方案

### 云函数身份验证

微信云开发的云函数在被小程序调用时，微信系统自动注入调用方的 `openid`，后端通过 `cloud.getWXContext()` 获取，**无法被客户端伪造**。

```typescript
import cloud from 'wx-server-sdk'

cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })

export const main = async (event: any) => {
  // openid 由微信系统自动注入，客户端无法伪造
  const { OPENID } = cloud.getWXContext()
  
  if (!OPENID) {
    // 非小程序调用（如直接 HTTP 调用）时为空
    return { code: 1002, message: '未登录' }
  }
  
  // 后续业务用 OPENID 作为用户标识
}
```

### 权限校验规则

| 操作 | 校验条件 |
|------|---------|
| 创建房间 | openid 有效 + consent_status=true |
| 加入房间 | openid 有效 + consent_status=true |
| 打卡 | openid 是该房间的活跃参与者 |
| 查看排行榜 | openid 是该房间的参与者（含已退出） |
| 删除账号 | openid 有效（无需其他条件） |
| 更新用户信息 | openid 有效 |

## 二、用户隐私保护

### 敏感数据分级

| 数据 | 级别 | 处理方式 |
|------|------|---------|
| 体重值（初始/每日） | **高敏感** | 不出现在日志；API 不返回他人绝对体重 |
| 打卡照片 | **高敏感** | 私有存储；临时 URL（2h）；不公开展示给其他成员 |
| openid | 内部标识 | 不对外 API 暴露；不出现在响应 body；日志可记录 |
| 微信昵称/头像 | 一般 | 正常使用，排行榜展示 |

### 日志脱敏规则

```typescript
// 错误示范 — 绝对禁止
console.log('用户打卡：', { openid, weight: 70500, roomId })

// 正确写法 — 脱敏日志
console.log('用户打卡成功', {
  roomId,
  type: 'weight',
  // weight 不打印，openid 截断
  openid: OPENID.substring(0, 6) + '***'
})
```

### API 隐私规则

- 排行榜：只返回 `weightLossPercent`（减重百分比），不返回 `weight`（绝对体重）
- 个人详情：只有本人可查自己的绝对体重历史
- 他人打卡照片：不对外提供访问接口（只有发起人可在管理端查阅）

### 数据删除合规

用户触发"删除账号"：
1. 同步删除所有个人数据（users / participants / checkins / score_snapshots）
2. 异步删除云存储照片（失败不影响主流程，后台补删）
3. 不保留任何形式的留存副本
4. 房间记录和排行榜历史：匿名化处理（nickname 改为"已注销用户"）

## 三、防作弊设计

### 3.1 体重打卡防作弊

| 校验项 | 规则 | 处理方式 |
|--------|------|---------|
| 体重合理性 | 30kg ≤ weight ≤ 300kg | 拒绝，返回 3003 |
| 与初始体重偏差 | 偏差 > 40%（如 80kg 初始，提交 < 48kg 或 > 112kg）| 拒绝，返回 3004 |
| 单日体重变化 | 与前一次打卡体重差 > 5kg | 拒绝，返回 3004 |
| 打卡日期 | 服务端按北京时间计算当天日期，不信任客户端传入 | 强制覆盖 |
| 照片必填 | 体重打卡必须上传照片 fileID | 缺失则返回 1001 |

### 3.2 打卡防重复（幂等设计）

```
DB 层：checkins 集合上 (room_id, openid, date, type) 唯一索引
服务层：写入前先查已有记录

体重打卡：
  有记录 → upsert 覆盖，code=0（允许当日修改）

饮食/运动打卡：
  有记录 → 直接返回现有状态，code=0（幂等，不报错）
```

### 3.3 房间操作防重

- 邀请码唯一索引，碰撞则重新生成（最多 3 次）
- 加入房间：`participants(room_id, openid)` 唯一索引，DB 层天然防重

## 四、输入校验规范

每个云函数都用 `joi` 做入参校验，校验失败统一返回 `code=1001`。

**通用规则：**

```typescript
import Joi from 'joi'

// roomId 校验（每个需要 roomId 的函数都用这个）
const roomIdSchema = Joi.string().pattern(/^[a-zA-Z0-9_-]{8,32}$/).required()

// 体重校验
const weightSchema = Joi.number().min(30).max(300).precision(1).required()

// 日期校验
const dateSchema = Joi.string().pattern(/^\d{4}-\d{2}-\d{2}$/).required()
```

## 五、AppSecret 与密钥管理

| 密钥 | 存放位置 | 禁止行为 |
|------|---------|---------|
| 微信 AppSecret | 云函数环境变量 | 不进代码库，不写死在代码里 |
| 微信支付证书（L2） | 云函数环境变量 / 密钥管理服务 | 同上 |
| 云数据库无密钥 | 云开发内部 IAM 管控 | 不暴露外部 |

**环境变量配置方式（云开发）：**
```
微信开发者工具 → 云开发控制台 → 云函数 → 环境变量
```

## 六、合规 Checklist

### 个人信息保护法要求

- [x] 首次使用显示隐私授权弹框，用户主动同意后才收集体重数据
- [x] 提供数据删除入口（`user-delete` 接口）
- [x] 不采集非必要字段（手机号、身份证、GPS 位置）
- [x] 照片仅用于打卡存证，不用于其他用途
- [x] 排行榜不展示绝对体重（只展示减重百分比）
- [x] 体重等敏感字段不出现在日志

### 微信小程序审核要求

- [x] 服务类目选"体育"或"生活服务"，不选"医疗健康"
- [x] 不使用绝对化减肥效果用语（"一定能瘦""保证减重"等）
- [x] 展示免责声明：「本工具仅供个人健身记录，不替代医疗建议」
- [x] 上线前提交《用户隐私保护指引》

## 七、安全应急响应

| 场景 | 处置流程 |
|------|---------|
| 发现用户体重数据泄露 | 立即通知 v_juvnliu → 评估影响范围 → 关闭相关接口 → 通知受影响用户 |
| 发现 SQL/NoSQL 注入 | 停用受影响云函数 → 紧急修复 → 重新上线 → 复盘 |
| 恶意刷接口 | 云开发控制台限制该 openid 的调用频率 → 封禁异常账号 |
| 照片存储异常访问 | 检查临时 URL 生成逻辑 → 确认私有存储配置 |
