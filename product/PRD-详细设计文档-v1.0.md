## 一、项目概述

### 1.1 背景

创始人与朋友三伏天约好减肥 PK，借 vibecoding 热潮从 0 构建微信小程序，解决「**熟人减肥 PK 缺乏公正计量和激励机制**」的真实需求。L1 自用版目标：你 + 朋友能跑完一次完整 PK 闭环。

### 1.2 产品定位

| 维度 | 说明 |
|------|------|
| 产品类型 | 微信小程序（原生开发，TypeScript） |
| 核心场景 | 熟人圈（2-5人）减肥 PK，有公正计量 + 激励机制 |
| 上线时间 | 2026-08-01 L1 自用版 |
| 北极星指标 | PK 完成率 ≥ 80%（所有参与者完成打卡至结算） |

### 1.3 明确不做（L1）

> 以下内容**不在本版本范围内**，任何新增需求一律写入 v1.1 backlog：

- 陌生人组队、社区 UGC、押金/奖励、商业化
- AI 识别（体重/食物）、硬件对接（蓝牙秤/HealthKit）
- 食物热量数据库、运动课程
- 精确地理位置采集

## 二、用户角色与用户故事

### 2.1 角色定义

| 角色 | 描述 | 权限 |
|------|------|------|
| **发起人** | 创建 PK 房间，设定规则，邀请朋友 | 创建房间、踢人（S4）、解散房间（需二次确认）|
| **参与者** | 接受邀请，登记初始体重，每日打卡 | 打卡、查看排行榜、退出房间 |

> 同一用户可以是不同 PK 房间的发起人或参与者（允许同时参加多个 PK，DR-005）

### 2.2 核心用户故事

| # | 角色 | 故事 | 验收标准 |
|---|------|------|----------|
| US-01 | 发起人 | 我想创建一个 PK 房间并设定规则，以便邀请朋友参加 | 能填写房间名+周期+开始日期，生成邀请链接/小程序码 |
| US-02 | 参与者 | 我想通过微信分享链接加入 PK | 点击链接进入小程序，展示规则，确认加入 |
| US-03 | 参与者 | 我想登记初始体重（含拍照留存） | 手动输入+拍照（时间水印），提交后不可修改 |
| US-04 | 参与者 | 我想每天打卡（体重+饮食+运动） | 三项打卡，体重当日可多次提交取最后一次 |
| US-05 | 所有人 | 我想随时看到排行榜 | T+1 00:05 更新，展示积分+减重% |
| US-06 | 所有人 | 我想在 PK 结束时看到结算报告和战报海报 | 自动结算，生成海报，支持保存+分享 |
| US-07 | 发起人 | 我想解散 PK 房间 | 解散后房间变为 DISSOLVED，成员可查看历史 30 天 |

### M1 用户体系

#### 1.1 微信一键登录

**流程**：

```
用户打开小程序
  → wx.login() 获取 code
  → 后端用 code 换取 session_key + openid
  → wx.getUserProfile() 获取昵称/头像（需用户手动点击授权）
  → 后端创建/更新用户记录
  → 返回自定义 JWT token
  → 前端存入 wx.setStorageSync('token', ...)
```

**字段采集**：

| 字段 | 来源 | 是否必填 | 说明 |
|------|------|----------|------|
| openid | wx.login code 换取 | 是 | 唯一用户标识，后端存储 |
| nickName | wx.getUserProfile | 是 | 用于排行榜展示 |
| avatarUrl | wx.getUserProfile | 是 | 用于排行榜展示 |
| 体重 | 用户手动输入 | PK时必填 | 敏感信息，单独同意 |

**不采集**：真实姓名、身份证、手机号、精确地理位置、饮食详情

#### 1.2 隐私授权弹框（首次必须）

- 触发时机：**首次打开小程序**（onLaunch 判断 hasAgreedPrivacy 本地标记）
- 内容：《敏感个人信息处理说明》单独同意弹框
  - 说明体重属于敏感个人信息
  - 用途仅限本 PK 记录，不公开展示给其他成员绝对值
- 用户必须点击"同意"才能继续使用，无"跳过"
- 同意后设置本地标记 + 后端记录同意时间戳

#### 1.3 免责声明

- 首次使用时展示，内容：「本工具仅供个人健身记录，不替代医疗建议」
- 「我的」页面常驻入口可再次查看

#### 1.4 数据删除入口

- 路径：「我的」→「设置」→「删除账号及所有数据」
- 操作：需二次确认弹框 → 后端软删除所有数据（标记 deleted_at）→ 退出登录
- 数据删除范围：用户记录 + 所有打卡数据 + 体重记录 + 上传照片（COS 文件同步删除）

### M2 PK 房间

#### 2.1 创建房间

**页面字段**：

| 字段 | 类型 | 校验规则 | 说明 |
|------|------|----------|------|
| 房间名 | 文本 | 1-20字，非空 | 如「三伏天减肥战」|
| PK 周期 | 数字 | 7-90 天 | 用户选择天数 |
| 开始日期 | 日期选择器 | 不早于今天 | 支持当天或未来 |
| 人数上限 | （L1 不设） | 无校验 | DR-003-D2/DR-005 已定：L1 不设上限 |

**房间状态机**：

```
待开始（PENDING）
  → [开始日期到达] → 进行中（ACTIVE）
  → [周期结束] → 已结束（ENDED）
  → [发起人解散] → 已解散（DISSOLVED）
```

**房间 ID 生成**：6位字母数字随机码（如 `A3K9PX`），用于邀请链接

#### 2.2 邀请机制

- 邀请链接格式：`https://mp.weixin.qq.com/xxx?roomCode=A3K9PX`
- 小程序码：调用微信 API 生成，存储于 COS
- 分享方式：`wx.shareAppMessage`（转发给朋友/群）
- 邀请有效期：房间处于 PENDING/ACTIVE 状态时链接有效；ENDED/DISSOLVED 后链接跳转提示「PK 已结束」

#### 2.3 加入房间流程

```
扫码/点链接 → 进入加入PK页
  → 展示：房间名 + PK周期 + 发起人 + 当前参与人数 + 积分规则说明
  → 用户点击「加入」
  → 录入初始体重（见 2.4）
  → 加入成功 → 进入 PK 详情页
```

#### 2.4 录入初始体重（关键流程）

**步骤详细**：

```
Step 1：输入体重
  - 手动输入（单位 kg，精度 0.1，范围 30-300kg）
  - 系统弹出强提示：「此体重为最终基准，提交后不可修改，请确认」

Step 2：拍照留存（必填）
  - 调用 wx.chooseMedia（sourceType: ['camera']，禁止 album）
  - 前端叠加水印：昵称 + 当前时间（YYYY-MM-DD HH:mm:ss）
  - 展示预览，用户确认

Step 3：提交
  - 图片通过 wx.uploadFile → /api/upload（后端中转存 COS）
  - 后端记录 weight_history（is_initial=true）
  - 提交成功后不可修改（需联系产品组长人工处理）
```

**体重校验**：

| 场景 | 处理 |
|------|------|
| < 30kg 或 > 300kg | 提示「请输入合理体重（30-300kg）」，不允许提交 |
| 照片上传失败 | 提示重试，不允许跳过 |
| 网络异常 | 数据本地暂存，恢复后自动重传 |

### M3 每日打卡

#### 3.1 打卡页面布局

```
打卡页（今日 2026-07-14）
├── 体重打卡卡片
│   ├── 输入框（今日体重 kg）
│   └── 拍照按钮（必填时）
├── 饮食打卡卡片
│   ├── 文字描述输入框（5-200字，必填）
│   └── 图片上传（可选，1-3张）
├── 运动打卡卡片
│   ├── 文字描述输入框（5-200字，必填）
│   └── 图片上传（可选，1-3张）
└── 提交按钮
```

> **DR-003-D1 已定**：饮食/运动打卡为「文字描述（必填，5-200字）+ 图片（可选，1-3张）」，非早期「二选一按钮」设计。

#### 3.2 体重打卡规则

| 规则 | 内容 |
|------|------|
| 是否强制 | 体重打卡**可选**（不强制每天填写） |
| 若填写 | 必须拍照留存（调用相机，禁止相册选图） |
| 水印内容 | 昵称 + 时间（YYYY-MM-DD HH:mm:ss） |
| 当日多次提交 | **允许**，以**最后一次**为准参与积分计算（DR-003 / MVP清单 M4 统一口径）|
| 打卡认定 | 当日有任意一次体重提交，即视为打卡成功（积分计算用最后一次） |
| 截止时间 | 当日 23:59 |

#### 3.3 饮食打卡规则

| 规则 | 内容 |
|------|------|
| 文字描述 | 必填，5-200 字 |
| 图片 | 可选，最多 3 张，可来自相机或相册（饮食/运动不限制相册） |
| 当日多次提交 | **允许**，取最后一次 |

#### 3.4 运动打卡规则

| 规则 | 内容 |
|------|------|
| 文字描述 | 必填，5-200 字 |
| 图片 | 可选，最多 3 张，可来自相机或相册 |
| 当日多次提交 | **允许**，取最后一次 |

#### 3.5 打卡提醒（S1）

- 默认时间：每日 20:00
- 方式：微信订阅消息（需用户提前订阅）
- 触发条件：当日 20:00 时，检测未完成体重打卡的成员，推送提醒
- 用户可自定义提醒时间（S1 实现，不阻塞 M 功能）

### M5 积分计算规则

#### 5.1 综合积分公式（最终版）

```
综合积分（满分100分）=
  减重积分 × 70%
  + 打卡积分 × 15%
  + 饮食打卡积分 × 8%
  + 运动打卡积分 × 7%
```

#### 5.2 各维度得分计算

**减重积分**（0-100分）：

```
减重积分 = min((初始体重 - 最新体重) / 初始体重 × 100 × K, 100)
```

- `初始体重`：加入时录入的初始值（全程固定基准，DR-002）
- `最新体重`：最近一次有效体重打卡数值（当日多次提交取最后一次）
- `K`：放大系数，暂定 K=6（减重10% → 60分，减重≈17% → 满分100分）
  > ⚠️ K 值待 v_juvnliu 7/16 前最终确认（当前按 K=6 开发，可配置化）

**打卡积分**（0-100分）：

```
打卡积分 = 已打卡天数 / 用户加入后实际经过天数 × 100
```

**饮食打卡积分**（0-100分）：

```
饮食打卡积分 = 饮食打卡完成天数 / 用户加入后实际经过天数 × 100
```

**运动打卡积分**（0-100分）：

```
运动打卡积分 = 运动打卡完成天数 / 用户加入后实际经过天数 × 100
```

> **DR-006-Q8 已定**：三个维度分母统一为「用户加入 PK 后的实际经过天数」，而非 PK 总天数。  
> 例：第3天加入21天PK，第21天结算时分母是 19（第3天到第21天）。  
> 迟加入者三个维度都能拿满分，公平性优先。

#### 5.3 积分精度规范

| 规范 | 说明 |
|------|------|
| 后端存储 | 保留 4 位浮点精度 |
| 前端展示 | 保留 2 位小数，四舍五入（DR-003-Q1）|
| 并列处理 | 不跳号（1,1,2），DR-003-Q2 |

#### 5.4 示例计算（K=6）

假设：A 用户初始体重 80kg，当前体重 75kg，已过 14 天中打卡 12 天，饮食 10 天，运动 11 天

```
减重积分 = min((80-75)/80 × 100 × 6, 100) = min(37.5, 100) = 37.50
打卡积分 = 12/14 × 100 = 85.71
饮食积分 = 10/14 × 100 = 71.43
运动积分 = 11/14 × 100 = 78.57

综合积分 = 37.50×70% + 85.71×15% + 71.43×8% + 78.57×7%
         = 26.25 + 12.86 + 5.71 + 5.50
         = 50.32 分
```

### M6 PK 结算

#### 6.1 自动结算触发

- **触发条件**：PK 周期结束日的次日 00:05（与每日积分更新任务合并）
- 结算以截止日 23:59 前最后一次有效数据为准（DR-003-Q3）
- 结算不可撤销，结算后房间状态变为 ENDED

#### 6.2 结算详情页

| 展示内容 | 说明 |
|----------|------|
| 最终排名 | 按综合积分降序，不跳号 |
| 各成员成绩卡 | 初始体重→结束体重（本人可见绝对值）、减重量、减重%、总打卡天数 |
| 本人排名详情 | 各维度得分明细 |

#### 6.3 战报海报生成

- **技术**：wx.createCanvasContext / Canvas 2D API
- **海报内容**：
  - 减肥PK小程序 Logo
  - 用户昵称 + 头像
  - PK 房间名称 + 时间周期
  - 最终排名（第 X 名）
  - 减重百分比（不展示绝对体重值，隐私保护）
  - 综合积分
  - 二维码/小程序码（引流）
- **操作**：一键保存到相册（wx.saveImageToPhotosAlbum）+ 分享到微信好友

## 五、异常处理规则

| 场景 | 处理方式 | 相关用例 |
|------|----------|----------|
| 今日已打卡，再次进入打卡页 | 体重打卡：展示已录入数据，允许覆盖（取最新一次）；完全无重复提交限制 | CHECKIN-007 |
| 体重输入 < 30kg 或 > 300kg | 提示「请输入合理体重（30-300kg）」，不允许提交 | CHECKIN-002 |
| 照片上传失败 | 提示重试，不允许跳过（照片是必填项，体重打卡时） | CHECKIN-005 |
| PK 未开始就尝试打卡 | 提示「PK 尚未开始，请等待 YYYY-MM-DD 开始日期」 | PERIOD-001 |
| 发起人意外退出房间 | 提示「发起人离开后房间将解散」，需二次确认；确认后房间状态 → DISSOLVED | EXIT-005 |
| 网络异常（打卡时） | 打卡数据本地暂存，网络恢复后自动重传（当日有效期内） | NET-001 |
| 房间已解散，成员尝试打卡 | 展示只读模式，提示「该 PK 已于 XX 解散，历史数据保留至 YYYY-MM-DD」| EXIT-006 |
| PK 结束日 23:59 后打卡 | 打卡入口关闭，提示「PK 已结束，等待结算中」 | PERIOD-007 |
| 邀请链接对应房间已结束 | 跳转并提示「PK 已结束，感谢参与」 | ROOM-008 |
| 初始体重填错，要求修改 | 提示「初始体重不可自助修改，请联系产品组长处理」，不提供修改入口 | WEIGHT-003 |

## 七、接口约定总览

### 7.1 通用规范

- 基础路径：`/api/v1/`
- 认证：`Authorization: Bearer <token>`（JWT，有效期 7 天，支持续期）
- 响应格式：
  ```json
  {
    "code": 0,         // 0=成功，非0=错误
    "message": "ok",
    "data": {}
  }
  ```
- 错误码：

| 错误码 | 说明 |
|--------|------|
| 0 | 成功 |
| 1001 | 未登录/Token 过期 |
| 1002 | 权限不足 |
| 2001 | 参数校验失败 |
| 2002 | 资源不存在 |
| 3001 | 体重范围错误 |
| 3002 | PK 未开始，不能打卡 |
| 3003 | PK 已结束 |
| 5000 | 服务器内部错误 |

### 7.2 核心接口列表

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | /auth/wx-login | 微信登录（换 token）|
| GET | /users/me | 获取当前用户信息 |
| DELETE | /users/me | 删除账号及数据 |
| POST | /rooms | 创建 PK 房间 |
| GET | /rooms/:roomCode | 获取房间信息 |
| POST | /rooms/:roomCode/join | 加入房间（含初始体重）|
| DELETE | /rooms/:roomId | 解散房间（房主） |
| POST | /rooms/:roomId/checkin/weight | 体重打卡 |
| POST | /rooms/:roomId/checkin/diet | 饮食打卡 |
| POST | /rooms/:roomId/checkin/exercise | 运动打卡 |
| GET | /rooms/:roomId/rank | 获取排行榜 |
| GET | /rooms/:roomId/settle | 获取结算结果 |
| POST | /upload | 上传图片（后端中转 COS）|

> 详细接口文档见 `backend/` 文件夹（斯斯负责输出）
