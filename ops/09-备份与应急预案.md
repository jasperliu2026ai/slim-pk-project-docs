# 09 · 备份策略与应急预案

> 版本：v1.0 | 2026-07-14

## 1. 备份策略总览

| 数据类型 | 备份方式 | 频率 | 保留期 | 恢复 RTO |
|----------|----------|------|--------|----------|
| MySQL 全量 | 云数据库自动备份 | 每天 02:00 | 7 天 | < 30 分钟 |
| MySQL 增量 | Binlog | 实时 | 7 天 | < 10 分钟（PITR）|
| Redis | AOF + RDB | AOF 实时，RDB 每小时 | 3 天 | < 5 分钟 |
| COS 用户数据 | 跨区复制（按需开启）| 实时同步 | 永久 | < 1 分钟 |
| 应用配置 | Git + SSM | 每次变更 | 永久 | < 5 分钟 |
| 系统配置 | Git 管理 | 每次变更 | 永久 | < 10 分钟 |

## 2. 故障分级与响应

| 级别 | 定义 | 典型示例 | 响应时限 | 通报范围 |
|------|------|----------|----------|----------|
| P0 | 全站不可用 / 支付异常 | API 全量 5xx、支付卡单 | 立即，5分钟内止血 | 老板 + 全团队 |
| P1 | 核心功能降级 | 打卡失败率 > 5%、排行榜不更新 | 15 分钟内响应 | 技术团队 |
| P2 | 性能劣化 / 次要功能 | API 慢、头像加载失败 | 1 小时内 | 运维 + 后端 |
| P3 | 资源预警 | 磁盘 80%、慢查询增加 | 工作时间处理 | 运维 |

## 3. 典型故障应急预案

### P0 - API 全量不可用

```
T+0   告警触发，值班人确认影响范围
T+1   SSH 登录 CVM：pm2 list / pm2 logs --err
T+2   进程崩溃 → pm2 restart slim-pk-api
      代码 Bug  → 执行回滚脚本 rollback.sh
T+5   健康检查通过，通知全员「已恢复」
T+30  业务指标稳定，启动故障复盘
```

### P0 - 数据库宕机

```
T+0   确认 DB 实例状态（腾讯云控制台）
T+2   高可用版自动主从切换（30-60 秒）
T+5   自动切换完成 → pm2 reload slim-pk-api（触发应用重连）
T+10  若未恢复 → 提腾讯云工单 + 评估降级只读模式
T+60  DB 恢复后 → 核查数据一致性（打卡/支付数据完整性）
```

### P1 - Redis 宕机

```
T+0   确认 Redis 主从状态（控制台）
T+2   主从版自动故障转移（约 30 秒）
T+5   恢复后触发缓存预热（访问排行榜接口）
T+10  若长时间未恢复 → 降级：排行榜降级直查 MySQL + Nginx 限流
```

### P1 - COS 上传失败

```
T+0   查看 https://status.cloud.tencent.com 确认服务状态
T+2   COS 故障 → 临时关闭图片必传校验（feature flag）
T+5   COS 恢复 → 开启校验，通知用户可重新上传
T+30  故障期间未上传的打卡记录人工核查
```

### P1 - 支付回调异常

```
⚠️ 涉及资金，必须谨慎，双人操作

T+0   查看回调日志：grep "payment.callback" /opt/slim-pk/app/shared/logs/*.log
T+2   微信商户后台确认支付平台侧订单状态
T+5   回调未到 → 手动触发查单接口同步状态
T+10  需人工补录 → DBA + 运维双人操作，留存操作记录
T+30  复盘回调链路，加强幂等保护
```
