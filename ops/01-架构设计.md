## 2. 技术栈选型

| 层次 | 选型 | 选择理由 |
|------|------|----------|
| 小程序前端 | 微信小程序原生 + TypeScript | 与微信生态深度集成，无需 Webview |
| 后端框架 | Node.js + Express | 团队现有技术栈，零改造成本 |
| 云服务商 | 腾讯云 | 微信生态亲和，支付/消息推送集成最顺滑 |
| 数据库 | TencentDB MySQL 8.0 高可用版 | 托管运维、自动备份、主从容灾 |
| 缓存 | 腾讯云 Redis 5.0 主从版 | 排行榜热点数据，打卡状态缓存 |
| 对象存储 | 腾讯云 COS | 打卡照片、称重截图、活动素材存储 |
| CDN | 腾讯云 CDN | COS 绑定 CDN，静态资源加速 |
| 负载均衡 | 腾讯云 CLB | 两台 CVM 跨可用区，无单点 |

## 3. 设计原则

1. **云托管优先**：数据库、缓存、存储全部使用云托管服务，不自建
2. **多可用区冗余**：CVM 至少 2 台跨可用区，规避机房级故障
3. **数据安全隔离**：数据库/Redis 子网不对公网暴露，只内网通信
4. **用户上传中转**：打卡照片走后端中转上传 COS，前端不直接持有 COS 密钥
5. **可回滚优先**：任何变更先灰度，有回滚方案才上线

## 4. 数据流

### 4.1 打卡流程
```
用户拍照 → 小程序合成水印 → wx.uploadFile 到 /api/v1/checkin/upload
→ 后端写 COS（临时密钥）→ 返回 COS URL → 入库 MySQL → 更新 Redis 打卡状态
```

### 4.2 排行榜流程
```
T+1 00:05 定时任务 → 拉 MySQL 当日数据 → 计算减重积分 → 写 Redis 排行榜
→ 客户端读 Redis 排行榜（GET /api/v1/rank）
```

### 4.3 支付流程
```
用户发起支付 → 后端调微信统一下单 → 返回预支付参数 → 小程序唤起支付
→ 微信异步通知 → 后端回调入库 → 幂等校验
```
