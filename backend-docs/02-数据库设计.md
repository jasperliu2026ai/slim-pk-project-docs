# 数据库设计

> 版本：v1.0 | 更新：2026-07-14 | 维护：斯斯【后端】  
> 数据库类型：微信云数据库（文档型，兼容 MongoDB 查询语法）

---

## 一、集合总览

| 集合名 | 说明 | 预估记录数（L1） |
|--------|------|-----------------|
| `users` | 用户基本信息 | < 100 |
| `pk_rooms` | PK 房间配置与状态 | < 20 |
| `participants` | 房间参与者（含初始体重） | < 200 |
| `checkins` | 每日打卡记录 | < 5000（21天×5人×3类型） |
| `score_snapshots` | 积分日快照（T+1 写入） | < 500 |
| `settlements` | PK 最终结算结果 | < 100 |

---

## 二、集合详细设计

### 2.1 users — 用户

```javascript
{
  _id: string,              // 云数据库自动生成
  openid: string,           // 微信 openid，全局唯一，不对外展示
  nickname: string,         // 微信昵称（首次授权获取）
  avatar_url: string,       // 微信头像 URL
  consent_status: boolean,  // 隐私同意状态（false 不可打卡）
  consent_at: Date | null,  // 同意时间
  created_at: Date,
  updated_at: Date
}
```

**索引：**
- `openid`（唯一索引）— 登录查询、防重复注册

**注意事项：**
- `openid` 不写入日志，不对外 API 暴露
- `avatar_url` 可能因用户换头像过期，排行榜展示时前端应做降级处理
- `consent_status = false` 时，`checkin-submit` 拒绝打卡并返回错误引导授权

---

### 2.2 pk_rooms — PK 房间

```javascript
{
  _id: string,
  name: string,             // 房间名称，用户自定义，最大 20 字
  creator_openid: string,   // 发起人 openid
  start_date: string,       // 'YYYY-MM-DD'，北京时间
  end_date: string,         // 'YYYY-MM-DD'，北京时间
  total_days: number,       // end_date - start_date + 1，由后端计算写入
  status: string,           // 'pending' | 'active' | 'ended'
  invite_code: string,      // 6 位随机邀请码（A-Z0-9），唯一
  created_at: Date,
  updated_at: Date
}
```

**索引：**
- `invite_code`（唯一索引）— 通过邀请码加入查询
- `creator_openid` — 查询"我创建的"房间
- `status` — 筛选进行中/已结束

**状态流转：**
```
pending → active（start_date 到达，定时任务激活）
active  → ended（end_date 结束，pk-settle 触发结算）
```

**业务规则：**
- L1 不限制人数上限（DR-003-D2），`max_members` 字段保留但不校验
- 邀请码碰撞概率：36^6 ≈ 21 亿种，生产环境几乎不会碰撞；碰撞时重新生成

---

### 2.3 participants — 房间参与者

```javascript
{
  _id: string,
  room_id: string,                // 关联 pk_rooms._id
  openid: string,                 // 参与者 openid
  initial_weight: number,         // 初始体重，单位：克（整数）
                                  // 例：70.5kg 存为 70500
  initial_photo_file_id: string,  // 云存储 fileID（初始体重拍照）
  joined_at: Date,
  is_active: boolean              // false = 已退出/被踢（积分冻结）
}
```

**索引：**
- `(room_id, openid)`（唯一复合索引）— 防重复加入
- `room_id` — 查房间所有参与者

**关键约束：**
- `initial_weight` 一经写入，服务层硬禁 UPDATE（代码级别 + 注释说明）
- `is_active = false` 后：积分冻结（score_snapshots 停止写入）、排行榜标记"已退出"、排名置末位
- 体重以**克（整数）**存储，避免浮点精度问题；展示时除以 1000 转 kg

---

### 2.4 checkins — 每日打卡记录

```javascript
{
  _id: string,
  room_id: string,          // 关联 pk_rooms._id
  openid: string,
  date: string,             // 'YYYY-MM-DD'，北京时间的打卡日期
  type: string,             // 'weight' | 'diet' | 'exercise'

  // type='weight' 时有值
  weight: number | null,            // 当日体重，克（整数）
  weight_photo_file_id: string | null, // 打卡照片 fileID

  // type='diet' 时有值
  diet_done: boolean | null,        // true=已控制饮食，false=未完成

  // type='exercise' 时有值
  exercise_done: boolean | null,    // true=已运动，false=未完成

  created_at: Date,
  updated_at: Date          // 体重打卡允许当日覆盖，更新此字段
}
```

**索引：**
- `(room_id, openid, date, type)`（唯一复合索引）— 幂等核心，防重复打卡
- `(room_id, date)` — 积分定时任务批量拉取某日所有打卡
- `(openid, room_id)` — 个人历史打卡曲线查询

**幂等策略（体重打卡可覆盖）：**
```
体重打卡：
  - 当日已有记录 → upsert 覆盖（取最新体重值）
  - 返回 code=0，不报错

饮食/运动打卡：
  - 当日已有记录 → 直接返回已有数据（幂等），不报错
  - 返回 code=0，data 中 dietDone/exerciseDone 为当前状态
```

**日期计算规范：**
- 打卡日期以**北京时间（UTC+8）**为准
- 服务端用 `new Date()` 获取 UTC 时间，加 8 小时换算为北京时间日期字符串
- 不依赖客户端传入的日期（防作弊）

### 2.5 score_snapshots — 积分日快照

```javascript
{
  _id: string,
  room_id: string,
  openid: string,
  snapshot_date: string,    // 'YYYY-MM-DD'，快照对应日期（T+1 写的是前一天的积分）

  // 原始分项数据（用于战报展示）
  current_weight: number,       // 克，最新体重
  initial_weight: number,       // 克，入组时初始体重（冗余存储，防 participants 被修改）
  weight_checkin_days: number,  // 累计体重打卡天数（有体重值的）
  diet_done_days: number,       // 累计饮食打卡完成天数
  exercise_done_days: number,   // 累计运动打卡完成天数
  elapsed_days: number,         // PK 已过天数（快照日 - 开始日 + 1）

  // 分项积分（×100 整数存储，展示时 /100 得两位小数）
  weight_loss_score: number,    // 减重分项 × 70，×100 存储
  checkin_score: number,        // 打卡分项 × 15，×100 存储
  diet_score: number,           // 饮食分项 × 8，×100 存储
  exercise_score: number,       // 运动分项 × 7，×100 存储
  total_score: number,          // 四项之和，×100 存储

  created_at: Date
}
```

**索引：**
- `(room_id, openid, snapshot_date)`（唯一复合索引）— upsert 幂等写入
- `(room_id, snapshot_date)` — 排行榜查询（某房间某日所有人的积分）

**设计说明：**
- 快照冗余存储，排行榜查询不需要实时计算，O(1) 读取
- `×100 整数`存储，彻底规避浮点精度问题
- 展示给用户时：`total_score / 100`，保留 2 位小数，四舍五入

---

### 2.6 settlements — 结算结果

```javascript
{
  _id: string,
  room_id: string,
  openid: string,

  // 最终数据
  final_weight: number,             // 克，PK 结束日的最新体重
  initial_weight: number,           // 克，入组初始体重
  weight_loss_grams: number,        // 减重克数（可为负，表示增重）
  weight_loss_pct: number,          // 减重百分比，×100 整数
  total_checkin_days: number,
  total_diet_done_days: number,
  total_exercise_done_days: number,
  total_pk_days: number,            // PK 总天数

  final_score: number,              // 最终积分，×100 整数
  rank: number,                     // 最终排名（1 = 第一名，并列不跳号）
  total_participants: number,       // 结算时的参与总人数

  settled_at: Date
}
```

**索引：**
- `(room_id, openid)`（唯一复合索引）— 防重复结算
- `room_id` — 结算页拉取整个房间所有人的战绩

## 三、数据安全规范

| 字段 | 处理方式 |
|------|----------|
| `initial_weight` / `weight` | 不出现在日志；展示时只展示减重%，不展示绝对体重 |
| 打卡照片 `fileID` | 私有存储，查看时用 `wx.cloud.getTempFileURL` 获取 2 小时临时链接 |
| `openid` | 不对外展示；后端内部使用；日志可记录（非直接敏感）|
| `session_key` | 不存储、不返回前端 |

## 四、积分计算示例

**场景**：PK 21 天，某用户已过 10 天

| 数据 | 值 |
|------|----|
| 初始体重 | 80,000 克（80kg）|
| 当前最新体重 | 77,000 克（77kg）|
| 体重打卡天数 | 8 天 |
| 饮食打卡完成天数 | 7 天 |
| 运动打卡完成天数 | 5 天 |
| elapsed_days | 10 |
| K 值 | 待确认（PRD 暂定 K=6）|

**以 K=6 计算：**

```
减重百分比 = (80000 - 77000) / 80000 × 100 = 3.75%
减重积分   = min(3.75 × 6, 100) = 22.5 分
减重分项   = 22.5 × 0.70 = 15.75 分

打卡率     = 8 / 10 × 100 = 80%
打卡分项   = 80 × 0.15 = 12.0 分

饮食完成率 = 7 / 10 × 100 = 70%
饮食分项   = 70 × 0.08 = 5.6 分

运动完成率 = 5 / 10 × 100 = 50%
运动分项   = 50 × 0.07 = 3.5 分

综合积分   = 15.75 + 12.0 + 5.6 + 3.5 = 36.85 分
存储值     = 3685（×100 整数）
展示值     = 36.85 分
```

> K 值待 v_juvnliu 确认（见 [08-待确认事项.md](./08-待确认事项.md)）
