## 二、影响接口设计的待确认项

### C1 🟡 体重打卡：允许当日覆盖 vs 仅一次

**背景**：
- MVP 清单 M4 写"体重可多次提交，以最后一次为准"
- PRD M3 异常处理写"今日已打卡，不允许重新提交"

**两者直接矛盾。**

**我的建议**：允许覆盖（取最新值），理由：用户测量误差常见，当天修正更合理；防作弊靠图片存证，不靠禁止覆盖。

**当前代码**：按"允许覆盖"实现（upsert），等确认后若要改为"禁止覆盖"需调整逻辑。

**等待**：菲菲 + v_juvnliu 确认

---

### C2 🟡 打卡积分分母：PK 总天数 vs 已过天数

**背景**：
- PRD M5：打卡积分分母 = PK 总天数；饮食/运动分母 = 已过天数
- DR-003：全部用"总周期天数"

**两种分母的效果差异**：

| 分母 | 第 3 天打卡率（打卡 3 次） | PK 结束时打卡率（打卡 18/21 天）|
|------|--------------------------|--------------------------------|
| 总天数（21）| 14.3%（看起来低）| 85.7% |
| 已过天数（3）| 100%（看起来好）| 85.7%（相同）|

**我的建议**：打卡积分分母用总天数（全程激励），饮食/运动分母用已过天数（当下激励）；与 PRD M5 一致。

**等待**：菲菲确认，截止 7/15

---

### C3 🟡 积分公式：PRD M5 vs DR-004

**背景**：
- PRD M5：减重积分 = `min((初始-最新)/初始 × 100 × K, 100)`，用**单次最新体重**
- DR-004：减重积分用**近 3 日均重**

**当前代码**：按 PRD M5（单次最新体重）实现，理由：简单、直观、与用户预期一致；3 日均重降低体重波动影响但增加计算复杂度。

**等待**：菲菲确认以哪个为准，截止 7/15

---

### C4 🟡 结算时间：23:59 vs 00:00

**背景**：
- PRD M6：PK 结束日 23:59 自动结算
- DR-003-Q3：结束日打卡有效，PK 在 00:00 结算

**当前实现**：定时任务 score-calc 在每日 00:05 运行；PK 结束日的 00:05 计算前一天（end_date）的积分并触发结算。即**实际结算时间为 end_date 次日 00:05**，与 DR-003-Q3 对齐。

**等待**：菲菲确认以 DR-003 为准，截止 7/15
