## 一、云函数部署

### 1.1 部署方式

微信云开发（CloudBase）的云函数通过微信开发者工具一键上传部署，无需手动配置服务器。

```bash
# 微信开发者工具 → 云开发 → 云函数 → 右键上传并部署
# 或使用 CLI：
npx @cloudbase/cli functions deploy --name auth-login --env slim-pk-prod-xxxx
```

### 1.2 环境配置

| 环境 | EnvID | 用途 |
|------|-------|------|
| 开发 | `slim-pk-dev-xxxx` | 本地联调 |
| 生产 | `slim-pk-prod-xxxx` | 线上用户 |

> ⚠️ EnvID 由韦韦【运维】创建后同步。严禁用开发环境 EnvID 跑生产流量。

### 1.3 部署前检查清单

```
[ ] TypeScript 编译通过：npm run build
[ ] 单测全绿：npm test
[ ] 环境变量已配置（见下方配置项清单）
[ ] 云函数内存/超时配置已核对
[ ] config.json 定时触发器配置正确（score-calc / pk-settle）
```

### 1.4 云函数资源配置

| 函数名 | 内存 | 超时 | 并发 | 说明 |
|--------|------|------|------|------|
| `auth-login` | 256MB | 5s | 默认 | 高频调用 |
| `checkin-submit` | 256MB | 10s | 默认 | 含图片校验 |
| `leaderboard-get` | 128MB | 5s | 默认 | 读快照，轻量 |
| `score-calc` | 512MB | 60s | 1 | 定时任务，全量计算 |
| `pk-settle` | 512MB | 60s | 1 | 结算任务，串行执行 |
| 其余函数 | 256MB | 10s | 默认 | 通用配置 |

## 二、定时触发器

### 2.1 积分计算（score-calc）

```json
{
  "triggers": [
    {
      "name": "score-calc-daily",
      "type": "timer",
      "config": "5 0 * * *"
    }
  ]
}
```

> 每天 00:05 触发（CST），计算前一日截止的积分快照。

### 2.2 PK 结算（pk-settle）

```json
{
  "triggers": [
    {
      "name": "pk-settle-daily",
      "type": "timer",
      "config": "10 0 * * *"
    }
  ]
}
```

> 每天 00:10 触发，检查当日是否有 PK 到期，若有则触发结算写入 `settlements` 集合。

### 2.3 订阅消息提醒（msg-remind）

```json
{
  "triggers": [
    {
      "name": "remind-nightly",
      "type": "timer",
      "config": "0 20 * * *"
    }
  ]
}
```

> 每日 20:00 检查当日未打卡用户并推送订阅消息。

## 三、数据库索引规划

> 云数据库索引在「云开发控制台 → 数据库 → 集合 → 索引」中配置。

| 集合 | 索引字段 | 类型 | 说明 |
|------|----------|------|------|
| `users` | `openid` | 唯一 | 登录查询 |
| `pk_rooms` | `invite_code` | 唯一 | 邀请码加入 |
| `pk_rooms` | `status` | 普通 | 定时任务扫 ACTIVE 房间 |
| `participants` | `(room_id, openid)` | 唯一复合 | 防重复加入 |
| `participants` | `openid` | 普通 | 查询我参与的房间 |
| `checkins` | `(room_id, openid, date, type)` | 唯一复合 | 防重复打卡（核心幂等） |
| `checkins` | `(room_id, date)` | 普通 | 定时任务按日批量查 |
| `score_snapshots` | `(room_id, openid, snapshot_date)` | 唯一复合 | 防重复写快照 |
| `score_snapshots` | `(room_id, snapshot_date)` | 普通 | 排行榜拉取 |
| `settlements` | `(room_id, openid)` | 唯一复合 | 结算数据唯一 |

## 四、监控与告警

### 4.1 云函数内置监控

在「微信开发者平台 → 云开发 → 监控」中可查看：
- 调用次数、成功率、错误率
- P50 / P95 / P99 延迟
- 内存使用情况

### 4.2 关键监控指标

| 指标 | 告警阈值 | 处理方式 |
|------|---------|--------|
| `checkin-submit` 错误率 | > 1% | 立即排查，5 分钟内响应 |
| `score-calc` 执行失败 | 任意失败 | 立即告警，手动补跑 |
| `pk-settle` 执行失败 | 任意失败 | 立即告警，手动触发结算 |
| 云函数超时 | 连续 3 次 | 检查数据库慢查询 |
| 云数据库容量 | > 80% | 提前扩容 |

### 4.3 日志查看

```bash
# 云开发控制台 → 云函数 → 日志
# 或使用 CLI：
npx @cloudbase/cli functions log --name checkin-submit --tail
```

**关键日志格式（结构化）：**

```
[auth-login] openid=o_xxxx action=login isNewUser=true cost=234ms
[checkin-submit] openid=o_xxxx roomId=room_xxx type=weight weight=68500g cost=456ms
[score-calc] roomCount=3 userCount=12 cost=8234ms date=2026-07-14
[pk-settle] roomId=room_xxx settled=true participantCount=5 cost=1234ms
```

## 五、故障处理手册

### 5.1 score-calc 定时任务失败

**现象**：排行榜数据未更新，停留在昨日快照。

**处理步骤**：
1. 查云函数日志，定位失败原因
2. 修复问题后，手动调用：
   ```javascript
   wx.cloud.callFunction({ name: 'score-calc', data: { manualDate: '2026-07-14' } })
   ```
3. 确认 `score_snapshots` 集合已写入正确数据
4. 在群里说明补跑情况，@菲菲【产品】知会

### 5.2 打卡接口报 5000

**处理步骤**：
1. 查 `checkin-submit` 日志，看完整堆栈
2. 常见原因：
   - 云数据库连接超时 → 等待或重试
   - 唯一索引冲突但未处理 → 检查幂等逻辑
   - fileID 非法 → 前端上传路径不对
3. 若影响用户，在小程序内发公告

### 5.3 排行榜数据异常

**现象**：积分显示 0 分或排名错乱。

**排查步骤**：
1. 查 `score_snapshots` 最新一条记录（云数据库 → score_snapshots → 按 snapshot_date 倒序）
2. 检查 `total_score` 是否为 ×100 整数（前端展示时需 /100）
3. 检查前端是否正确处理了并列排名逻辑

### 5.4 结算未触发

**现象**：PK 已到期但结算数据未生成。

**处理步骤**：
1. 手动调用 `pk-settle` 云函数（可传 `roomId` 指定房间）
2. 检查 `settlements` 集合是否已写入
3. 若已写入但前端未显示，检查 `pk_rooms.status` 是否已更新为 `ended`

## 六、数据备份

云数据库自动备份策略：
- **自动备份**：每天凌晨 2:00（微信云开发默认，保留 7 天）
- **手动导出**：控制台 → 数据库 → 导出 → JSON 格式

**重要数据保护原则**：
- `settlements` 集合：结算完成后不可修改，只增不删
- `participants.initial_weight`：服务层硬禁 UPDATE，如需修改走专项流程并记录日志
- 所有积分相关变更需有操作日志

## 七、版本发布流程

```
1. 本地开发 → 单测通过（npm test）
2. 提交 PR / commit → 代码自审
3. 部署到开发环境 → 前后端联调通过
4. 部署到生产环境（微信开发者工具或 CLI）
5. 冒烟测试：完整走一遍 PK 创建→打卡→排行榜→结算流程
6. 观察 30 分钟监控（错误率、延迟）
7. 如有问题：重新部署上一个版本（云函数版本管理 → 回滚）
```
