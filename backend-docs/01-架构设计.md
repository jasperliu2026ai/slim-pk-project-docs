# 后端架构设计

> 版本：v1.0 | 更新：2026-07-14 | 维护：斯斯【后端】

---

## 一、总体架构

```
微信小程序客户端
  wx.cloud.callFunction()  |  wx.cloud.uploadFile()
         |                          |
  云函数（CloudBase）            云存储（COS）
  Node.js 16 + TypeScript       打卡照片 / 战报海报
  12 个业务函数                  私有存储，临时 URL
         |
  云数据库（文档型）
  6 个集合 + 复合索引
  T+1 积分快照
```

### 架构特点

1. **无服务器（Serverless）**：后端全部跑在云函数上，无常驻进程，按调用计费
2. **文档型数据库**：云数据库类似 MongoDB，用集合（Collection）替代表
3. **直接鉴权**：云函数自动获取调用方 `openid`，无需单独实现 JWT
4. **图片直传**：前端通过 `wx.cloud.uploadFile` 直传云存储，不走云函数中转

## 二、技术选型

| 层 | 选型 | 版本 | 说明 |
|---|---|---|---|
| 运行时 | Node.js | 16+ | 云函数支持的最高稳定版 |
| 语言 | TypeScript | 4.x | strict 模式，禁 any |
| 云函数 SDK | wx-server-sdk | latest | 云数据库 / 云存储 / 云函数互调 |
| 参数校验 | joi | ^17 | 每个云函数入参统一校验 |
| 日志 | console.log + 云日志 | - | L1 用内置 console，L2 接高级日志 |
| 定时任务 | 云函数定时触发器 | cron | 每日 00:05 跑积分结算 |

## 四、公共规范

### 4.1 统一响应格式

```typescript
// 成功
interface SuccessResponse<T = unknown> {
  code: 0
  message: 'ok'
  data: T
}

// 失败
interface ErrorResponse {
  code: number    // 非 0，见错误码表
  message: string // 用户可读的中文描述
  detail?: string // 仅开发环境返回
}
```

### 4.2 错误码规范

| 错误码 | 含义 |
|--------|------|
| `0` | 成功 |
| `1001` | 参数校验失败 |
| `1002` | 未登录（openid 为空） |
| `1003` | 无权限操作 |
| `2001` | 房间不存在 |
| `2002` | 已在房间中，禁止重复加入 |
| `2003` | PK 已结束，不可操作 |
| `2004` | PK 未开始，不可打卡 |
| `3001` | 今日已打卡（体重可覆盖，饮食/运动幂等） |
| `3002` | 打卡时间不在 PK 周期内 |
| `3003` | 体重值超出合理范围（30-300kg） |
| `3004` | 体重变化异常（与初始偏差 > 40%，或单日变化 > 5kg）|
| `5000` | 系统内部错误 |

### 4.3 云函数标准模板

```typescript
import cloud from 'wx-server-sdk'
import Joi from 'joi'

cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })

const schema = Joi.object({
  roomId: Joi.string().required(),
  type: Joi.string().valid('weight', 'diet', 'exercise').required(),
})

export const main = async (event: any, context: any) => {
  // 1. 获取调用方 openid（云函数自动注入，无法伪造）
  const { OPENID } = cloud.getWXContext()
  if (!OPENID) {
    return { code: 1002, message: '未登录' }
  }

  // 2. 参数校验
  const { error, value } = schema.validate(event)
  if (error) {
    return { code: 1001, message: error.message }
  }

  try {
    // 3. 业务逻辑
    const result = {} // 实际业务
    return { code: 0, message: 'ok', data: result }
  } catch (err: any) {
    console.error('[function-name] error:', err)
    return { code: 5000, message: '系统内部错误' }
  }
}
```

## 五、数据流向

### 5.1 打卡主链路

```
用户拍照 → 前端 Canvas 合成时间水印
    ↓
wx.cloud.uploadFile → 云存储（返回 fileID）
    ↓
wx.cloud.callFunction('checkin-submit', {
  roomId, type: 'weight', weight: 68.5, photoFileId
})
    ↓
云函数：校验参数 → 查房间状态 → 防重复（DB 唯一索引）
    → 写 checkins 集合 → 返回今日打卡汇总
    ↓
前端更新打卡状态 UI
```

### 5.2 积分结算链路

```
每日 00:05 定时触发 score-calc
    ↓
查所有 status=ACTIVE 的房间
    ↓
for each 房间 / for each 参与者：
  查前一天打卡记录 → 计算分项积分
  → 写 score_snapshots（upsert，幂等）
    ↓
排行榜查询时直接读 score_snapshots 最新快照
```

### 5.3 排行榜查询

```
前端调 leaderboard-get { roomId }
    ↓
云函数：查 score_snapshots，where roomId + 最新 snapshot_date
    → 排序 total_score DESC
    → 处理并列排名（相同分数 → 相同名次，下一名次不跳号：1,1,2）
    ↓
返回排行榜数组（不暴露绝对体重，仅展示减重%）
```
